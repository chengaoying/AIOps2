## 1. 产品概述

### 1.1 产品定位

| 属性 | 描述 |
|------|------|
| 产品名称 | AIOps 大数据智能诊断平台 |
| 产品类型 | B端数据诊断 Dashboard |
| 目标用户 | 大数据平台运维工程师、数据工程师、DevOps 团队 |
| 核心场景 | 凌晨作业失败时，快速诊断问题根因 |
| 核心价值 | "30秒内给出根因分析和修复建议" |

### 1.2 设计原则

- **功能优先，简单易用**：减少运维压力，所有元素服务功能
- **实用主义**：视觉噪音归零，无渐变、无阴影、无装饰性图形
- **快速高效**：信息密度高但层次清晰，一眼识别关键信息

## 2. 用户故事

### 2.1 核心场景

```
作为 运维工程师
我希望在 凌晨收到告警后
能够 在30秒内了解作业失败原因和修复建议
以便 快速响应，减少故障时长
```

### 2.2 用户痛点

| 痛点 | 现状 | 期望 |
|------|------|------|
| 排查耗时长 | 登录多系统，手动查日志 | 一键诊断 |
| 跨平台复杂 | YARN/Hive/Spark/Flink 各不同 | 统一视图 |
| 根因不明确 | 日志分散，难以定位 | AI 自动分析 |
| 修复方案模糊 | 需要自行研究 | 给出具体建议 |

## 3. 功能规格

### 3.1 核心功能

#### F1: Dashboard 首页
- 今日概览统计（诊断总数、成功数、失败数、成功率）
- 平台分布可视化（YARN/Spark/Hive/Flink 占比）
- 最近诊断列表（带状态色条和快捷操作）

#### F2: 作业诊断
- 输入：作业 ID 或作业日志
- 诊断流程：自然语言解析 → 知识库检索（向量+关键词）→ LLM 推理 → 输出结果
- 输出：根因分析 + 修复建议（带置信度和风险等级）

#### F3: 诊断历史
- 按时间/平台/状态筛选
- 支持查看历史诊断详情

#### F4: AI 助手
- 自然语言查询作业问题
- 支持多轮对话

#### F5: 系统配置
- 用户管理（角色：管理员/普通用户）
- 集群配置（多集群管理）
- 系统配置（阈值、告警规则）

### 3.2 功能优先级

| 优先级 | 功能 | 原因 |
|--------|------|------|
| P0 | 作业诊断 | 核心价值所在 |
| P0 | Dashboard 首页 | 第一入口 |
| P1 | 诊断历史 | 复盘需求 |
| P1 | AI 助手 | 增强诊断能力 |
| P2 | 系统配置 | 运维必备 |

## 6. 技术架构

### 6.1 核心架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 消息队列 | 去掉 Kafka，Collector 直写 StarRocks | 简化架构，降低复杂度 |
| 诊断策略 | 知识库+LLM 混合（知识库检索+LLM 推理） | 可维护性强，覆盖面广，支持持续学习 |
| LLM 安全 | 三层防护（输入验证+结构化 Prompt+输出验证） | 确保 AI 输出安全可控 |
| 高可用 | Collector/API 无状态微服务 | 便于扩展和故障恢复 |
| 知识库 | Vector 存储+全文检索 | 高效相似度匹配 |
| 自愈系统 | Phase 2 | Phase 1 聚焦诊断核心价值 |

### 6.4 核心数据流

#### 6.4.1 数据采集流
```
YARN/Hive/Spark/Flink
    → Collector Plugin (REST API/SDK)
    → Memory Queue (10000条上限)
    → WAL (磁盘持久化)
    → Batch Writer (每5秒或1000条)
    → StarRocks
```

#### 6.4.2 知识库构建流
```
文档来源
    ├── 内部文档 (Confluence/Wiki)
    ├── 官方文档 (Hadoop/Spark/Hive/Flink)
    └── 运维记录 (历史诊断记录)
            │
            ▼
    文档采集器 (Doc Collector)
            │
            ▼
    文档解析器 (PDF/MD/HTML/Doc)
            │
            ▼
    LLM 分析提炼器 (LLM Analyzer)
    ├── 实体提取 (错误类型、平台、根因、解决方案)
    ├── 结构化知识 (知识卡片)
    └── 向量化 (Embedding)
            │
            ▼
    知识库存储 (Knowledge Store)
    ├── Vector Store (向量检索)
    └── Full-text Index (全文检索)
```

#### 6.4.3 诊断请求流
```
User Request
    → NL Query Parser (自然语言→结构化查询)
    → Context Builder (获取作业链、日志、相似案例)
    → Knowledge Base Retrieval (知识库检索)
            │
            ▼
    ┌─────────────────────────────────┐
    │  知识库检索 (Vector + Keyword)  │
    │  1. 错误模式 → 向量相似度匹配  │
    │  2. 关键词 → 全文检索         │
    │  3. 融合排序 (RRF)             │
    └─────────────────────────────────┘
            │
            ▼
    LLM 推理 (LLM Reasoner)
    ├── 构建诊断 Prompt (作业信息+知识上下文)
    └── 生成诊断结果 (根因+建议)
            │
            ▼
    Response (根因+建议+置信度+参考知识)
```

#### 6.4.4 告警触发流
```
异常检测 (Trend Analysis)
    → Alert Manager
    → 渠道分发 (钉钉/飞书/企微/邮件)
    → Sidebar Notification
```

### 6.5 知识卡片结构

```go
type KnowledgeCard struct {
    ID          string    `json:"id"`
    Platform    string    `json:"platform"`     // YARN/HIVE/SPARK/FLINK
    ErrorType  string    `json:"error_type"`  // 错误类型
    ErrorPattern []string `json:"error_pattern"` // 错误模式（多种表述）

    // LLM提炼的结构化知识
    RootCause  string      `json:"root_cause"`   // 根因分析
    Suggestions []Suggestion `json:"suggestions"`   // 修复建议

    // 元数据
    Source     string      `json:"source"`       // 来源文档
    Confidence float64     `json:"confidence"`   // 置信度
    Tags       []string   `json:"tags"`         // 标签

    // 向量表示
    Embedding  []float32 `json:"embedding"`

    // 时间戳
    CreatedAt  time.Time  `json:"created_at"`
    UpdatedAt  time.Time  `json:"updated_at"`
}

type Suggestion struct {
    Action   string `json:"action"`    // 修复动作
    Risk     string `json:"risk"`     // 低/中/高
    Detail   string `json:"detail"`   // 详细说明
    Command  string `json:"command"`  // 可执行命令（如有）
}
```
