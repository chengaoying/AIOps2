## Vision

### 核心定位调整

**从被动诊断工具 → 主动AI助手**

原设计：作业失败后触发诊断，推送结果
新定位：用户可以随时问"为什么变慢了""哪些作业最耗资源"，系统主动回答

这是产品差异化的关键。让AIOps不只是一个监控告警工具，而是一个懂大数据运维的AI助手。

### 用户体验愿景

想象运维工程师的一天：

**早上9点** - 打开AIOps，看到Dashboard显示集群健康度90/100
- 主动问："昨晚有哪些作业失败了？"
- 系统答："3个作业失败，最严重的是daily_etl，OOM导致。已自动诊断，点击查看详情。"

**上午10点** - 收到钉钉告警"Spark作业变慢"
- 点击通知进入Chat界面，系统已经分析好：
  "spark_job_001执行时间从5分钟增加到15分钟。根因：数据量增长3倍但Executor未调整。建议：增加Executor从10个到30个。"

**下午3点** - 主动优化
- 问："哪些作业最耗资源？"
- 系统答："Top 3资源消耗作业：daily_etl (32GB内存)、user_analysis (28GB)、log_processing (24GB)。建议优化daily_etl的Shuffle操作。"

**下午5点** - 趋势分析
- 问："本周作业性能有什么变化？"
- 系统答："整体稳定。但user_analysis作业执行时间持续上升，从周一的10分钟到今天的18分钟。建议检查数据增长趋势。"

## 分期策略

### Phase 1: 元仓+诊断（AI驱动+AI交互）

**核心能力**：
1. **被动诊断**（已有设计）
   - 作业失败 → 自动诊断 → 推送结果
   - 规则引擎 + LLM混合诊断

2. **告警触发诊断**（已有设计）
   - 监控告警 → 自动诊断 → 钉钉/飞书通知
   - 统一运维仪表板

3. **AI交互查询**（新增）
   - 自然语言查询："为什么变慢了""哪些作业最耗资源"
   - 性能基线建立：记录作业历史执行时间/资源
   - 趋势分析：对比历史数据发现异常
   - 对话式UI：Chat界面 + 图表展示

**技术架构**：
```
现有架构：
Collector Agent → StarRocks → Diagnosis API → Dashboard

新增模块：

1. NL Query Parser（新增）
   用户问题 → LLM解析 → 生成SQL/API调用
   
   例："为什么变慢了？" → 
        SELECT duration_sec, memory_mb 
        FROM unified_job_view 
        WHERE job_id='xxx' 
        ORDER BY created_at DESC LIMIT 30

2. Performance Baseline Service（新增）
   定时计算（每小时）：
   - 每个作业的p50/p95执行时间
   - 每个作业的平均资源消耗
   - 存储到job_baseline表

3. Trend Analysis Engine（新增）
   对比逻辑：
   current_duration > baseline_p95 * 1.5 → 性能退化
   current_memory > baseline_avg * 2.0 → 资源异常
   
4. Chat UI（新增）
   复用React Dashboard，添加Chat组件
   输入框 + 对话历史 + 图表展示
```

**元仓核心能力**（已在架构评审中确认）：
- 统一元数据视图（unified_job_view）
- 作业级关联表（job_dependency）
- 资源使用趋势视图（mv_resource_trend）

**工作量估算**：

- CC+gstack：~2周（核心1.5周 + AI交互3天）

### Phase 2: SQLScan（事前预防）

**核心能力**：
1. **SQL规则引擎**
   - 性能检测：全表扫描、笛卡尔积、大表JOIN
   - 安全检测：SQL注入、权限绕过
   - 规范检测：命名规范、注释缺失

## Accepted Scope (Phase 1新增)

### AI交互查询能力

#### 1. NL Query Parser
- LLM理解用户意图
- 生成SQL查询或API调用
- 支持的查询类型：
  - 性能分析："为什么变慢了""执行时间趋势"
  - 资源分析："哪些作业最耗资源""内存使用排行"
  - 故障分析："最近有哪些失败""失败原因分布"
  - 趋势分析："本周性能变化""数据量增长趋势"

#### 2. Performance Baseline Service
- 定时任务（每小时）：
  ```sql
  INSERT INTO job_baseline
  SELECT 
    job_id,
    component,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY duration_sec) as p50_duration,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY duration_sec) as p95_duration,
    AVG(memory_mb) as avg_memory,
    AVG(vcores) as avg_vcores,
    DATE_FORMAT(NOW(), '%Y-%m-%d %H:00:00') as baseline_time
  FROM unified_job_view
  WHERE created_at >= NOW() - INTERVAL 7 DAY
  GROUP BY job_id, component;
  ```

#### 3. Trend Analysis Engine
- 异常检测逻辑：
  ```go
  type AnomalyDetector struct {
    baseline BaselineService
  }
  
  func (d *AnomalyDetector) DetectPerformanceAnomaly(job Job) *Anomaly {
    baseline := d.baseline.GetBaseline(job.ID)
    
    // 性能退化检测
    if job.Duration > baseline.P95Duration * 1.5 {
      return &Anomaly{
        Type: "performance_degradation",
        Severity: "high",
        Message: fmt.Sprintf("执行时间从%d秒增加到%d秒", 
                            baseline.P50Duration, job.Duration),
      }
    }
    
    // 资源异常检测
    if job.Memory > baseline.AvgMemory * 2.0 {
      return &Anomaly{
        Type: "resource_spike",
        Severity: "medium",
        Message: fmt.Sprintf("内存使用从%dMB增加到%dMB", 
                            baseline.AvgMemory, job.Memory),
      }
    }
    
    return nil
  }
  ```

## Technical Considerations

### Architecture
- **去掉Kafka**：Collector Agent直接写StarRocks（带WAL缓冲）
- **无状态微服务**：Collector Agent和Diagnosis API支持水平扩展
- **LLM安全防护**：Prompt Injection三层防护（输入验证+结构化Prompt+输出验证）

### Performance
- **批量写入**：Collector Agent每5秒或1000条批量写入
- **结果缓存**：Diagnosis API使用Redis缓存（TTL=1小时）
- **规则引擎**：前缀树索引，匹配时间<10ms
- **LLM并发控制**：令牌桶限流，10 QPS

### Cost Optimization
- **规则+LLM混合**：70%场景用规则（免费），30%用LLM（付费）
- **节省成本**：70%（从每月3000-15000元降到900-4500元）
