## Problem Statement

企业大数据平台运维面临的核心困境（聚焦YARN/Hive/Spark/Flink）：
- **日志分散**：YARN作业日志、Hive查询日志、Spark Application日志、Flink Job日志各自独立，故障排查需要在多个位置手动查找
- **定位困难**：作业失败的原因可能藏在资源不足、SQL错误、数据倾斜、Checkpoint失败等多个层面
- **AI能力缺失**：现有工具依赖规则匹配，无法理解上下文和业务语义，误报率高
- **经验传承难**：专家经验以文档或口口相传形式存在，无法系统化复用

**一句话痛点**：当凌晨3点YARN/Hive/Spark/Flink作业失败时，运维工程师需要在多个Web UI、日志文件中手动查找原因，而AI可以在30秒内给出根因分析和修复建议。

根据用户确认：

**核心价值主张**：将实践（一站式诊断+AI增强）产品化，面向有多大数据平台的企业提供开箱即用的诊断能力。

## Target User & Narrowest Wedge

### 核心用户画像
- **大数据平台运维工程师**：负责保障CDH/HDP集群稳定运行，熟悉Cloudera Manager或Ambari
- **数据工程师**：需要快速诊断Hive/Spark/Impala作业异常
- **DevOps团队**：负责HDFS、Hive元数据、YARN资源配置稳定性

### 一期MVP范围（元仓+诊断，AI增强）
**聚焦**：YARN、Hive、Spark、Flink作业日志及运行信息采集与诊断

最小可用版本必须包含：
1. **元数据仓库**：自动采集YARN/Hive/Spark/Flink作业元数据
2. **日志采集**：作业运行日志、错误信息、诊断信息
3. **AI诊断引擎**：基于大模型的根因分析（输入症状，输出原因+建议）
4. **可视化界面**：诊断结果展示+历史追溯

**不是一期**（二期再做）：
- SQLScan规则扫描
- Kafka/HDFS/HBase等扩展组件采集

- SQL质量门禁

## Approaches Considered

### Approach A: 全开源自建（Minimal MVP）
**技术栈**：StarRocks + Prometheus + LangChain + 开源LLM
- **优势**：无供应商锁定，运维可控
- **劣势**：需要自建大模型服务，推理成本高

### Approach B: SaaS化产品（推荐）
**技术栈**：云原生架构 + 多LLM适配 + API优先

```
优势：
✅ 云原生部署，支持K8S一键安装
✅ 多LLM后端适配（OpenAI/Claude/通义/DeepSeek）
✅ 租户隔离，支持私有化部署
✅ 参考Bilibili实践，经过验证的架构

劣势：
⚠️ 需要处理数据安全问题（客户可能不愿上云）
⚠️ 多引擎适配增加复杂度
```

### Approach C: 垂直行业切入
**技术栈**：聚焦某一行业（金融/电商）深度定制

```
优势：
✅ 客户更精准，需求更痛
✅ 可以做深度集成

劣势：
⚠️ 市场天花板低
⚠️ 定制化成本高
```

---

## Recommended Approach

**选择 Approach B（SaaS化产品）**，理由：
1. **参考架构经过验证**：Bilibili实践已证明SQLScan+StarRocks架构可行
2. **AI是核心竞争力**：SaaS模式便于接入最新LLM能力
3. **市场空间大**：不绑定行业，可服务所有有大数据平台的企业
4. **技术护城河**：元仓+诊断的规则积累是长期壁垒

#### 3.2 物化视图（加速诊断查询）

```sql
-- 作业失败趋势统计
CREATE MATERIALIZED VIEW mv_job_failure_stats
AS SELECT
    platform,
    DATE_FORMAT(created_at, '%Y-%m-%d %H') as hour,
    status,
    COUNT(*) as cnt
FROM job_meta
GROUP BY platform, DATE_FORMAT(created_at, '%Y-%m-%d %H'), status;

-- Top错误码统计
CREATE MATERIALIZED VIEW mv_top_error_codes
AS SELECT
    platform,
    error_code,
    COUNT(*) as cnt,
    AVG(duration_sec) as avg_duration
FROM job_meta
WHERE status = 'failed'
GROUP BY platform, error_code;
```

```yaml
# 诊断规则示例 - Phase 1聚焦的4个组件
rules:
  # ========== YARN规则 ==========
  - id: "yarn_oom"
    component: "yarn"
    condition: "diagnostic_info CONTAINS 'OutOfMemoryError' OR exit_code = 137"
    diagnosis:
      type: "memory_insufficient"
      cause: "Container内存配置不足或作业内存泄漏"
      evidence_patterns:
        - "memory needed: (\d+)MB"
        - "container killed due to memory"
      recommendations:
        - action: "增加YARN Container内存"
          risk: "low"

  - id: "yarn_container_launch_failed"
    component: "yarn"
    condition: "diagnostic_info CONTAINS 'Container launch failed'"
    diagnosis:
      type: "dependency_missing"
      cause: "作业依赖的本地资源或文件缺失"

  - id: "yarn_resource_contention"
    component: "yarn"
    condition: "final_status = 'FAILED' AND diagnostic_info CONTAINS 'queue'"
    diagnosis:
      type: "queue_resource_full"
      cause: "YARN队列资源耗尽"
      recommendations:
        - action: "检查队列资源使用情况"
        - action: "调整队列容量配置"

  # ========== Hive规则 ==========
  - id: "hive_semantic_error"
    component: "hive"
    condition: "error_message CONTAINS 'SemanticException'"
    diagnosis:
      type: "sql_semantic_error"
      cause: "SQL语法或语义错误"

  - id: "hive_serde_conflict"
    component: "hive"
    condition: "error_message CONTAINS 'SerDeException'"
    diagnosis:
      type: "table_schema_conflict"
      cause: "表SerDe配置与实际数据格式不匹配"

  - id: "hive_memory_error"
    component: "hive"
    condition: "error_message CONTAINS 'OutOfMemoryError' OR error_message CONTAINS 'GC overhead'"
    diagnosis:
      type: "memory_insufficient"
      cause: "Hive执行内存不足"

  # ========== Spark规则 ==========
  - id: "spark_executor_oom"
    component: "spark"
    condition: "failure_reason CONTAINS 'ExecutorLost' AND error_message CONTAINS 'OOM'"
    diagnosis:
      type: "executor_memory"
      cause: "Executor内存不足或数据倾斜"

  - id: "spark_stage_failed"
    component: "spark"
    condition: "end_state = 'FAILED' AND failure_reason CONTAINS 'Task'"
    diagnosis:
      type: "task_failure"
      cause: "Task执行失败，可能是数据问题或资源问题"

  - id: "spark_shuffle_error"
    component: "spark"
    condition: "error_message CONTAINS 'ShuffleMapStage' OR error_message CONTAINS 'shuffle'"
    diagnosis:
      type: "shuffle_failure"
      cause: "Shuffle阶段失败，可能是磁盘IO或网络问题"

  # ========== Flink规则 ==========
  - id: "flink_checkpoint_timeout"
    component: "flink"
    condition: "failure_reason CONTAINS 'Checkpoint expired' OR failure_reason CONTAINS 'checkpoint timeout'"
    diagnosis:
      type: "checkpoint_timeout"
      cause: "Checkpoint超时，数据量太大或状态后端配置不足"

  - id: "flink_task_manager_oom"
    component: "flink"
    condition: "error_message CONTAINS 'OutOfMemoryError' AND task_manager"
    diagnosis:
      type: "tm_memory"
      cause: "TaskManager内存不足"

  - id: "flink_kafka_connect_timeout"
    component: "flink"
    condition: "task_failures[].error CONTAINS 'TimeoutException' AND task_failures[].task CONTAINS 'Kafka'"
    diagnosis:
      type: "source_timeout"
      cause: "Kafka数据源读取超时"
```

#### 4.3 YARN/Hive/Spark/Flink特有诊断场景

| 场景 | 诊断类型 | 关键证据来源 | 规则覆盖 |
|------|----------|--------------|----------|
| **YARN Container OOM** | 资源不足 | AM日志 + 集群内存使用率 | ✅ yarn_oom |
| **YARN队列满** | 资源竞争 | 队列容量 + 等待作业数 | ✅ yarn_resource_contention |
| **Hive语义错误** | SQL错误 | 查询计划 + 错误信息 | ✅ hive_semantic_error |
| **Hive SerDe冲突** | Schema冲突 | 表定义 + 数据格式 | ✅ hive_serde_conflict |
| **Spark Executor OOM** | 数据倾斜 | Stage统计 + Executor日志 | ✅ spark_executor_oom |
| **Spark Shuffle失败** | IO瓶颈 | Shuffle数据量 + 磁盘使用 | ✅ spark_shuffle_error |
| **Flink Checkpoint超时** | 状态后端 | Checkpoint大小 + 超时配置 | ✅ flink_checkpoint_timeout |
| **Flink TM内存** | 资源不足 | TM内存使用 + 堆外内存 | ✅ flink_task_manager_oom |

## Module Partition

### Phase 1 MVP（本期实现）

### Module 1: AIOps Agent（采集模块）
**职责**：YARN/Hive/Spark/Flink作业日志及运行信息采集
- 基于REST API的插件化设计
- Phase 1支持：YARN、Hive、Spark、Flink
- 支持扩展：Kafka、HDFS、HBase等（Phase 2+）
- 暴露指标：采集延迟、处理速率、错误率

### Module 2: MetaStore（元仓模块）
**职责**：统一元数据存储和查询
- StarRocks + Iceberg冷热分层
- 支持Schema变更历史追踪
- 提供血缘分析和影响评估API

### Module 3: Diagnosis Engine（诊断模块）
**职责**：AI驱动的根因分析和修复建议
- 单LLM后端（通义千问）- MVP简化多LLM适配
- 规则引擎（基础）+ AI增强（核心差异化）
- **注：知识图谱和案例库为Phase 2功能**

### Module 4: Dashboard（前端模块）
**职责**：可视化界面和交互
- 诊断历史和追溯
- 实时告警和通知
- 报表和统计

### Phase 2（本期后实现）

### Module 5: SQLScan（SQL扫描模块）
**职责**：静态SQL代码质量检查
- DDL规范检查（表名、分区命名）
- SQL性能检测（Join顺序、全表扫描）
- 安全扫描（SQL注入、敏感字段访问）

> **注**：SQLScan是独立的产品模块，与运行时作业诊断互补。
